# 控制器问题

## ros2_control控制器故障诊断

解决控制器加载、运行和性能问题。


##  控制器问题概述

### 常见问题

```
□ 控制器无法加载
□ 控制器无法启动
□ 控制器运行异常
□ 轨迹执行失败
□ 性能下降
□ 控制器崩溃
```

### 基本检查

```bash
# 列出控制器状态
ros2 control list_controllers

# 检查硬件接口
ros2 control list_hardware_interfaces

# 查看日志
ros2 topic echo /rosout | grep -i error
```


##  控制器无法加载

### 症状

```bash
ros2 control load_controller left_arm_controller
# 返回: Failed to load controller
```

### 原因与解决

**原因1: 配置文件错误**

```bash
# 检查配置文件
cat ~/openarmx_robotstride_ws-cc/src/openarm_ros2/openarm_controllers/config/openarm_controllers.yaml

# 常见错误:
# - YAML语法错误(缩进)
# - 控制器类型拼写错误
# - 关节名称错误
```

**验证YAML:**
```bash
python3 -c "import yaml; yaml.safe_load(open('openarm_controllers.yaml'))"
```

**原因2: 控制器插件未安装**

```bash
# 检查是否安装
ros2 pkg list | grep joint_trajectory_controller

# 如未安装
sudo apt install ros-humble-joint-trajectory-controller
sudo apt install ros-humble-position-controllers
```

**原因3: 工作空间未编译**

```bash
cd ~/openarmx_robotstride_ws-cc
colcon build --packages-select openarm_controllers
source install/setup.bash
```


##  控制器无法启动

### 症状

```bash
ros2 control set_controller_state left_arm_controller start
# 控制器状态仍为inactive
```

### 原因与解决

**原因1: 硬件接口不可用**

```bash
# 检查硬件接口
ros2 control list_hardware_interfaces

# 应看到:
# command interfaces:
#   left_joint_1/position [available] [claimed]
#   ...

# 如果状态为[unavailable]:
# - 硬件接口未初始化
# - CAN未连接
# - 电机未使能
```

**解决:**
```bash
#  检查CAN
ip link show can0

#  启动CAN
python3 en_all_can.py

#  使能电机
python3 en_all_motors.py

#  重启控制器
ros2 launch openarmx_bringup openarm.launch.py
```

**原因2: 接口被其他控制器占用**

```bash
# 查看claimed状态
ros2 control list_hardware_interfaces

# 如果已被占用:
# 停止冲突的控制器
ros2 control set_controller_state <冲突控制器> stop
```

**原因3: 配置错误**

```python
# 检查控制器配置
# joints列表必须与硬件接口匹配

# 错误示例:
joints:
  - left_joint_1
  - left_joint_2
  - left_joint_8  # ✗ 不存在

# 正确:
joints:
  - left_joint_1
  - left_joint_2
  - left_joint_3
  - left_joint_4
  - left_joint_5
  - left_joint_6
  - left_joint_7
```


##  轨迹执行失败

### 症状

```
MoveIt规划成功,但Execute失败
或轨迹执行中断
```

### 诊断

```bash
# 监控控制器状态话题
ros2 topic echo /left_arm_controller/state

# 监控Action结果
ros2 topic echo /left_arm_controller/follow_joint_trajectory/result
```

### 常见失败原因

**原因1: 轨迹点超出限位**

```yaml
# 检查joint_limits.yaml
left_joint_1:
  has_position_limits: true
  min_position: -3.14
  max_position: 3.14
```

**监控代码:**
```python
import rclpy
from control_msgs.action import FollowJointTrajectory

def result_callback(future):
    result = future.result().result
    error_code = result.error_code

    if error_code == 0:
        print("✓ 轨迹执行成功")
    elif error_code == -1:
        print("✗ 失败: INVALID_GOAL")
    elif error_code == -2:
        print("✗ 失败: INVALID_JOINTS")
    elif error_code == -3:
        print("✗ 失败: OLD_HEADER_TIMESTAMP")
    elif error_code == -4:
        print("✗ 失败: PATH_TOLERANCE_VIOLATED")
    elif error_code == -5:
        print("✗ 失败: GOAL_TOLERANCE_VIOLATED")
```

**原因2: 时间戳问题**

```python
# 轨迹点time_from_start必须单调递增

# 错误:
point1.time_from_start = Duration(sec=1, nanosec=0)
point2.time_from_start = Duration(sec=0, nanosec=500000000)  # ✗

# 正确:
point1.time_from_start = Duration(sec=0, nanosec=0)
point2.time_from_start = Duration(sec=1, nanosec=0)  # ✓
```

**原因3: 路径容差超限**

```yaml
# openarm_controllers.yaml
left_arm_controller:
  ros__parameters:
    constraints:
      left_joint_1:
        trajectory: 0.1  # 路径容差(rad)
        goal: 0.05       # 目标容差(rad)
```

**调整:**
```yaml
# 放宽容差
trajectory: 0.2  # 从0.1增加
goal: 0.1        # 从0.05增加
```


##  性能问题

### 症状

```
- 控制延迟大
- 更新频率低
- CPU使用率高
- 运动不流畅
```

### 诊断工具

```bash
# 检查更新频率
ros2 topic hz /joint_states

# 应为100-200Hz
# 如果<50Hz,说明有性能问题

# 检查CPU
htop
# 查看controller_manager进程CPU使用率

# 检查延迟
ros2 topic delay /joint_states
```

### 性能优化

**优化1: 调整update_rate**

```yaml
# openarm_controllers.yaml
controller_manager:
  ros__parameters:
    update_rate: 100  # 降低频率减轻负担
```

**优化2: 降低state_publish_rate**

```yaml
left_arm_controller:
  ros__parameters:
    state_publish_rate: 50.0  # 从100降低
```

**优化3: 使用实时内核**

```bash
# 安装RT内核
sudo apt install linux-image-rt-amd64

# 重启选择RT内核

# 提高进程优先级
sudo chrt -f 80 -p $(pgrep controller_manager)
```


##  控制器崩溃

### 症状

```
- controller_manager进程退出
- 所有控制器变为inactive
- 无法控制机器人
```

### 查看崩溃日志

```bash
# ROS2日志
ros2 topic echo /rosout | grep -i fatal

# 系统日志
journalctl -u ros2* --since "10 minutes ago"

# 核心转储(如果启用)
coredumpctl list
coredumpctl info <pid>
```

### 常见崩溃原因

**原因1: 硬件接口异常**

```
CAN通信中断 → 硬件接口超时 → 崩溃

解决:
- 确保CAN稳定
- 添加异常处理
- 增加超时时间
```

**原因2: 内存泄漏**

```bash
# 监控内存
watch -n 1 "ps aux | grep controller_manager"

# 如果内存持续增长:
# - 检查代码是否有内存泄漏
# - 重启定期任务
```

**原因3: 配置冲突**

```
多个控制器尝试claim同一接口
→ 资源冲突 → 崩溃

解决:
- 检查控制器配置
- 确保无重复claim
```


##  诊断流程

### 系统化诊断

```
1. 检查Controller Manager状态
   ros2 service call /controller_manager/list_controllers
   ↓
2. 检查硬件接口
   ros2 control list_hardware_interfaces
   ↓
3. 检查控制器配置
   cat openarm_controllers.yaml
   ↓
4. 查看日志
   ros2 topic echo /rosout
   ↓
5. 测试单个控制器
   ros2 control load_controller <name>
   ↓
6. 监控性能
   ros2 topic hz /joint_states
```

### 诊断脚本

```python
#!/usr/bin/env python3
"""
控制器健康检查
"""
import rclpy
from rclpy.node import Node
from controller_manager_msgs.srv import ListControllers

class ControllerDiagnostic(Node):
    def __init__(self):
        super().__init__('controller_diagnostic')

        self.client = self.create_client(
            ListControllers,
            '/controller_manager/list_controllers'
        )

    def diagnose(self):
        """诊断控制器"""
        print("========== 控制器诊断 ==========\n")

        # 1. 检查Controller Manager
        if not self.client.wait_for_service(timeout_sec=5.0):
            print("✗ Controller Manager 不可用")
            return

        print("✓ Controller Manager 可用\n")

        # 2. 列出控制器
        request = ListControllers.Request()
        future = self.client.call_async(request)
        rclpy.spin_until_future_complete(self, future)

        result = future.result()

        if len(result.controller) == 0:
            print("✗ 未找到控制器")
            return

        print(f"找到 {len(result.controller)} 个控制器:\n")

        # 3. 检查每个控制器
        issues = []

        for ctrl in result.controller:
            status_icon = "✓" if ctrl.state == "active" else "✗"
            print(f"{status_icon} {ctrl.name}")
            print(f"  类型: {ctrl.type}")
            print(f"  状态: {ctrl.state}")

            # 检查问题
            if ctrl.state != "active":
                issues.append(f"{ctrl.name} 状态异常: {ctrl.state}")

            # 检查必需控制器
            if 'joint_state_broadcaster' in ctrl.name and ctrl.state != 'active':
                issues.append("joint_state_broadcaster 必须active")

            print()

        # 4. 总结
        if len(issues) == 0:
            print("✓ 所有控制器正常")
        else:
            print(f"✗ 发现 {len(issues)} 个问题:")
            for issue in issues:
                print(f"  - {issue}")

def main():
    rclpy.init()
    diagnostic = ControllerDiagnostic()
    diagnostic.diagnose()
    rclpy.shutdown()

if __name__ == '__main__':
    main()
```


##  常见错误信息

### 错误信息速查

| 错误信息 | 原因 | 解决 |
|---------|------|------|
| "Failed to load controller" | 配置错误/插件未安装 | 检查配置,安装插件 |
| "Controller already active" | 重复启动 | 先停止再启动 |
| "Hardware interface not available" | 硬件未连接 | 检查CAN,启动硬件 |
| "Joint names mismatch" | 关节名称不匹配 | 检查配置文件 |
| "Trajectory tolerance violated" | 超出容差 | 放宽容差或改善轨迹 |
| "Invalid goal" | 轨迹点超限 | 检查关节限位 |
| "Controller not configured" | 未配置 | 先configure再start |
